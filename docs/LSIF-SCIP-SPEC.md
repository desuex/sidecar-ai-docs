# LSIF / SCIP Specification

---

## 1. Purpose

This document defines how LSIF and SCIP can be integrated into the indexing subsystem.

LSIF (Language Server Index Format) and SCIP (Symbol Cross-reference Index Protocol) provide:

* Precomputed symbol indexes
* Cross-reference graphs
* Definition and reference mapping
* Language-aware symbol identity

They are especially useful for:

* Large repositories
* CI-generated indexes
* Offline analysis
* Monorepos
* Cross-workspace linking

They are optional but highly valuable.

---

## 2. Why LSIF / SCIP

LSP is interactive.

LSIF / SCIP are persistent.

LSP:

* Requires active server
* May be slow for large repos
* May depend on local environment

LSIF / SCIP:

* Precomputed
* CI-friendly
* Deterministic
* Portable
* Fast to load

They provide a stable semantic baseline.

---

```text id="o7x9ks"

Normalized Symbol Graph

Documentation Binding
```

They replace or augment LSP during indexing.

---

## 4. LSIF Overview

LSIF is:

* JSON-based graph format
* Vertex + edge model
* Encodes:

  * Symbols
  * Ranges
  * Definitions
  * References
  * Hover content

LSIF is commonly generated by:

* TypeScript
* Go
* Java
* C#
* Rust (via tools)

LSIF is:

* Large
* Verbose
* CI-generated

---

## 5. SCIP Overview

SCIP is:

* Compact binary or JSON format
* Designed as successor to LSIF
* More efficient encoding
* Stable symbol identifiers
* Strong language ecosystem support

SCIP provides:

* Symbol IDs
* Occurrence locations
* Definition mapping
* Relationship metadata

SCIP is preferred when available.

---

## 6. Integration Strategy

System must implement:

* LSIF importer
* SCIP importer
* Format normalization
* UID mapping

Import pipeline:

1. Load LSIF / SCIP file.
2. Parse symbol definitions.
3. Map external symbol ID → internal UID.
4. Store references.
5. Merge into persistent index.

---

## 7. Symbol Identity Mapping

LSIF/SCIP provide their own symbol IDs.

System must:

* Preserve original ID as metadata.
* Generate internal UID.
* Maintain bidirectional mapping.

Example mapping:

```json id="b0mv4n"
{
  "external_symbol": "scip:typescript mypkg/src/foo.ts#MyClass#",
  "internal_uid": "sym:ts:mypkg/src/foo.ts:MyClass"
}
```

UID generation must remain deterministic.

External IDs are not authoritative.

---

## 8. Merging Strategy

When both Tree-sitter + LSIF/SCIP exist:

* Tree-sitter provides structure.
* LSIF/SCIP provide semantic resolution.
* System reconciles both.

Priority rules:

* LSIF/SCIP definition mapping overrides heuristic matching.
* Tree-sitter structural fingerprinting remains primary for anchoring.
* LSP may provide incremental updates on top.

---

## 9. Partial Index Scenarios

In CI-only mode:

* Entire index built from LSIF/SCIP.
* No live LSP required.
* Documentation binding still works via UID.

In local mode:

* Import LSIF baseline.
* Apply incremental changes via Tree-sitter + LSP.

---

## 10. Performance Considerations

LSIF files can be large.

System must:

* Stream parse.
* Avoid loading entire file into memory if possible.
* Build incremental index.
* Support chunked processing.

SCIP is more compact and preferred for large repos.

---

## 11. Cross-Repository Linking

SCIP supports global symbol identifiers.

This enables:

* Cross-repository linking.
* Shared dependency mapping.
* External library resolution.

System must:

* Support external symbol references.
* Mark external UID scope.
* Avoid requiring full external source.

---

## 12. Refactor Considerations

LSIF/SCIP:

* Reflect state at generation time.
* Do not update automatically.
* Must be regenerated after major refactor.

System must:

* Detect mismatch between source hash and index.
* Invalidate outdated entries.
* Allow reimport.

Anchoring layer must not depend solely on LSIF range positions.

---

## 13. Limitations

LSIF / SCIP:

* Depend on language tooling.
* May be incomplete for dynamic languages.
* Do not provide structural fingerprint.
* May miss runtime-generated references.
* May not reflect partial code edits.

They enhance semantic accuracy but do not replace structural parsing.

---

## 14. Storage Requirements

Imported LSIF/SCIP data must be normalized into:

* Symbol table
* Reference graph
* External symbol registry
* Source hash mapping

Raw LSIF/SCIP files may optionally be cached.

---

## 15. Security Considerations

LSIF/SCIP import must:

* Treat files as untrusted input.
* Validate structure.
* Prevent injection via malformed symbol IDs.
* Avoid executing build scripts.

Import layer must be purely parsing logic.

---

## 16. Schema Versioning

LSIF and SCIP versions may change.

Importer must:

* Detect format version.
* Support known schema versions.
* Reject incompatible versions gracefully.

---

## 19. Summary

LSIF / SCIP provide:

* Persistent semantic index
* Accurate cross-reference mapping
* CI-friendly indexing
* Large-repo scalability
* Cross-repository linking

Tree-sitter provides:

* Structural parsing
* Refactor fingerprinting
* Incremental updates

LSP provides:

* Live semantic resolution

Together they form:

Persistent baseline + Structural resilience + Live updates

This combination enables scalable, refactor-aware documentation infrastructure.
